# v0.1.0-alpha release include/exclude manifest

Status: v0.1.0-alpha publication manifest. It authorizes only the requested alpha tag/release path; service restarts and existing-asset overwrites remain separately controlled.

## Matched workflow

The current release asset workflow is `.github/workflows/release-assets.yml`.

Expected asset names generated by the workflow:

```text
ga-admin-v0.1.0-alpha-windows-amd64.zip
ga-admin-v0.1.0-alpha-windows-amd64.zip.sha256
ga-admin-v0.1.0-alpha-windows-arm64.zip
ga-admin-v0.1.0-alpha-windows-arm64.zip.sha256
ga-admin-v0.1.0-alpha-darwin-amd64.zip
ga-admin-v0.1.0-alpha-darwin-amd64.zip.sha256
ga-admin-v0.1.0-alpha-darwin-arm64.zip
ga-admin-v0.1.0-alpha-darwin-arm64.zip.sha256
ga-admin-v0.1.0-alpha-linux-amd64.zip
ga-admin-v0.1.0-alpha-linux-amd64.zip.sha256
ga-admin-v0.1.0-alpha-linux-arm64.zip
ga-admin-v0.1.0-alpha-linux-arm64.zip.sha256
```

The workflow packages files from `dist/*` into a top-level directory named `ga-admin-v0.1.0-alpha-<goos>-<goarch>/`, then uploads `package/*.zip` and `package/*.sha256`.

## Include manifest

Each platform zip should include only runtime files needed by ordinary users:

| Path in zip | Windows | macOS/Linux | Purpose |
| --- | --- | --- | --- |
| `ga-admin.exe` | yes | no | Windows GUI executable. |
| `ga-admin` | no | yes | macOS/Linux executable. |
| `cmd/chat_worker.py` | yes | yes | Fixed relative worker path used by the Chat panel. |
| `README.txt` | yes | yes | Short package-local launch notes. |
| Embedded web UI in executable | yes | yes | React/Vite UI is embedded by the Go build and is not a separate source tree in the zip. |

Source repository changes intended for the release commit may include user-facing source, build scripts, docs, workflow files, and required static assets. Release review reports and temporary evidence are not runtime release assets.

## Exclude manifest

The following files and directories must not be included in release zips or treated as public release assets:

| Pattern or path | Reason |
| --- | --- |
| `config.local.json` | Local host/port/GA-root configuration. |
| `model_profiles.json` | Local model profile drafts and user-specific provider data. |
| `mykey.py` | Private API key and model configuration generated on the user's machine. |
| `.env`, `*.key`, tokens, cookies | Secrets or credentials. |
| `.git/`, `.github/` inside zips | Repository metadata and automation definitions are not runtime files. |
| `web/node_modules/` | Dependency cache; rebuilt by CI, not shipped. |
| Local `reports/` and `reports/_*_evidence/` | Review notes, gate logs, and troubleshooting artifacts. |
| Temporary backups such as `*.bak`, `*.tmp`, `*.orig` | Non-product artifacts. |
| Existing `dist/` content from a developer machine | Must be regenerated by the approved workflow or local packaging step before use. |
| Board keys, API keys, release tokens, browser cookies | Operational secrets; never part of release assets. |

Before publication, inspect the produced zip listing and sha256 sidecar. A clean package should not rely on `.gitignore`; it should be verified from the actual archive contents.

## Offline and error-state limitations

| Area | Offline or error limitation | Expected user-facing behavior | Release verification note |
| --- | --- | --- | --- |
| Files | File operations are limited to the configured GA root. Missing roots, denied paths, traversal attempts, and deleted files can still fail. | Show a clear empty/error state and require explicit confirmation for destructive actions. | Verify errors do not expose private absolute paths beyond what the local admin already configured. |
| Models | The app can draft and write local `mykey.py`, but cannot prove an external provider key, endpoint, quota, or model name works offline. | Validate obvious malformed fields before writeback and tell the user provider connectivity must be tested separately. | Ensure no API key is committed or packaged. |
| Schedule | Scheduled task editing depends on local OS permissions and command availability. Offline mode cannot prove future command success. | Show disabled/error states for missing task data and require review before changing commands. | Do not restart or mutate live tasks during documentation validation. |
| BBS | The BBS panel depends on the configured board URL/key and a reachable local/remote board. A disabled board or bad key may return empty/error states. | Display disabled, unauthorized, or connection-failed states without retry loops that spam the board. | Worker protocol posts remain operational metadata, not release assets. |
| Process and services | Process discovery and stop/start operations are OS- and permission-dependent. PIDs may become stale between display and action. | Re-check current state where possible and keep stop/restart behind dangerous-operation confirmation. | Do not stop live user services as part of release packaging checks. |
| Observability and health | Health/risk views summarize local state and may be partial when dependencies are absent. | Mark stale, missing, or partial data instead of presenting it as complete. | Acceptance should record the command or route used to collect evidence. |
| Version and update | Update discovery requires a published Release asset and matching `.sha256`. Alpha publication does not by itself approve formal `v0.1.0` promotion. | If assets are absent, mismatched, blocked by network, or checksum fails, show a recoverable error and do not replace the running binary. | Post-publication verification is required after the alpha workflow finishes. |
| TMWebDriver | Status depends on local Python packages, port `18766`, and extension files. Offline repair may only install from reachable package indexes. | Report missing dependencies and provide guarded repair/install actions. | Repair should not stop browser processes. |

## Release-owner approval boundary

Alpha publication has release-owner approval for the following boundary; anything outside it requires separate approval:

1. Final commit range and target branch.
2. Target tag `v0.1.0-alpha` for this alpha publication; formal `v0.1.0` remains separate.
3. Exact commit, tag, push, and Release commands.
4. Whether existing remote assets may be overwritten.
5. Final gate evidence and archive-content inspection.
6. Post-publication update-flow verification plan.
